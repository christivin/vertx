# Vertx 功能交付追踪

## 1. 使用规则

本文件用于把 `vertx-product-spec.md` 的功能清单落成逐项交付状态。后续实现必须遵守：

- 同一时间只允许一个最高优先级功能处于 `实现中`
- 每个功能都要经过：
  - 实现
  - 测试
  - 验证
  - 修改
  - 再验证
  - 完成
- 如果某个功能仍处于“只读骨架”“静态 mock”“缺少关键交互”的状态，不能标记为完成
- 功能完成后，才进入下一个同优先级功能

## 2. 状态说明

- `未开始`：还没有进入实现阶段
- `进行中`：已经开始实现，但未形成完整闭环
- `部分可用`：能展示或部分可操作，但还不满足真实可用
- `待验证`：代码和测试基本完成，等待联调或手动验证
- `已完成`：已实现、已测试、已验证、必要修正完成
- `延期`：明确不在当前阶段闭环，但保留在总清单中

## 3. 当前执行顺序

当前建议的顺序固定为：

1. F-007 会话详情 Realtime 主链路
2. F-011 接入管理
3. F-012 设置管理
4. F-009 知识源管理
5. F-010 自动化管理
6. F-005 运行详情
7. F-002 工作台总览
8. F-003 / F-004 / F-006 / F-013 细化补强
9. F-017 到 F-020 工程强化

原因：

- F-007 是 Vertx 与普通 controller 聚合方案拉开差异的核心
- F-011 / F-012 / F-009 / F-010 决定“每个模块都真实可用”是否成立
- F-005 和 F-002 决定整体产品是否形成完整闭环

## 5. 当前焦点功能卡

### 当前唯一激活功能

- 功能：F-007 会话详情 Realtime 主链路
- 状态：`待验证`
- 进入原因：
  - 它直接决定 Vertx 是否真的建立了区别于 controller 聚合文本的主聊天体验
  - 它也是审批、tool stream、run lifecycle 感知的承载页

### F-007 当前已知缺口

- 仍缺一次真实页面联调，确认 mock / gateway 两种 plane 下都符合预期
- 断线恢复与 history sync 已接入，但还需要一次人工验证确认不会误伤活跃 run
- 当前审批只做到“感知与展示”，还没有提交审批动作
- 虽然已有页面级自动化测试，但 gateway 真实连接场景还没有人工联调

### F-007 完成标准

- 路由 `sessions/:sessionId` 能正确驱动详情页
- 会话页能够展示基础 session 信息
- 发送任务后能看到本地用户消息、stream delta、tool 状态、审批状态
- 断线恢复和历史同步路径不破坏当前活跃流
- 至少完成类型检查、单测和一次手动页面验证

## 6. 执行要求

后续每完成一次阶段工作，都要更新本文件：

- 若开始实现某功能：修改“当前状态”和“下一步”
- 若新增测试：更新“测试”列
- 若完成手动验证：更新“验证”列
- 若修正问题：更新“修改”列
- 若再次通过验证：更新“再验证”列，并把功能改为 `已完成`
